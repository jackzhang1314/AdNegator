## 1. 产品概述

### 1.1 产品定义

gVisor E2B 是一个基于 gVisor 容器运行时的**代码执行沙盒平台**，提供与 E2B (E2B.dev) **100% API 兼容**的云端代码执行环境。

### 1.2 问题陈述

**当前痛点**：
- E2B 基于 AWS Firecracker，对基础设施要求高（需要裸金属或嵌套虚拟化）
- E2B 商业服务成本高昂，中小企业难以承受
- 自建代码执行沙盒缺乏安全隔离，容易被攻击
- 现有开源方案与主流容器生态割裂，运维复杂

**核心问题**：
> "如何用更低的成本和更简单的架构，提供与 E2B 同等安全性和兼容性的代码执行沙盒？"

### 1.3 解决方案

采用 **gVisor** 替代 Firecracker，在保持 E2B SDK/API 完全兼容的前提下：
- 降低基础设施要求（支持标准云虚拟机）
- 简化部署流程（原生 Docker/Kubernetes 支持）
- 降低运营成本（50-100ms 冷启动，节约资源）
- 保持安全隔离（系统调用级别隔离）

## 4. 核心价值主张

### 4.1 对比 E2B 的核心优势

| 维度 | E2B (Firecracker) | gVisor E2B | 优势说明 |
|------|-------------------|------------|----------|
| **部署要求** | 裸金属 / 嵌套虚拟化 | 标准云虚拟机 | 降低 IaaS 成本 60% |
| **冷启动** | 150-300ms | 50-100ms | 用户体验更优 |
| **容器编排** | 定制方案 | 原生 K8s | 运维复杂度降低 80% |
| **API 兼容** | E2B 原生 | 100% 兼容 | 零迁移成本 |
| **开源协议** | 商业闭源 | MIT/Apache 2.0 | 私有化部署无限制 |

### 4.2 安全性保障

- **系统调用隔离**：gVisor 拦截所有系统调用，恶意代码无法直接访问宿主机内核
- **资源限制**：CPU、内存、磁盘、网络全方位配额管理
- **网络隔离**：沙盒间网络默认隔离，可选白名单出站访问
- **文件系统隔离**：每个沙盒独立文件系统，无法访问宿主机文件

## 5. 功能需求

### 5.1 核心功能 (P0)

#### F1: 沙盒生命周期管理

**用户故事**：
> 作为开发者，我需要能够快速创建、连接、暂停、恢复和销毁代码执行沙盒。

**功能点**：
- [F1.1] 创建沙盒 (`POST /sandboxes`)
  - 支持模板 ID（预装环境）
  - 支持超时自动销毁
  - 支持自定义环境变量
  - 支持 metadata 标签

- [F1.2] 连接沙盒 (`POST /sandboxes/{id}/connect`)
  - 已暂停沙盒自动恢复
  - 返回 envd 访问令牌

- [F1.3] 暂停沙盒 (`POST /sandboxes/{id}/pause`)
  - 使用 CRIU 检查点技术
  - 暂停后释放 CPU 资源
  - 保留内存快照到持久化存储

- [F1.4] 销毁沙盒 (`DELETE /sandboxes/{id}`)
  - 清理容器
  - 清理检查点文件
  - 清理临时存储

- [F1.5] 列表查询 (`GET /sandboxes`)
  - 支持按状态筛选
  - 支持分页

**验收标准**：
- 创建到可用 < 2s (99th percentile)
- 暂停操作 < 3s (1GB 内存)
- 恢复操作 < 3s (1GB 内存)
- 销毁操作 < 1s

---

#### F2: 代码执行引擎

**用户故事**：
> 作为开发者，我需要在沙盒内执行代码，获取实时输出和退出状态。

**功能点**：
- [F2.1] 进程管理
  - 启动进程 (via envd gRPC)
  - 实时流式输出 (stdout/stderr)
  - 进程中断/杀死
  - 获取退出码

- [F2.2] 文件系统操作
  - 上传文件到沙盒
  - 从沙盒下载文件
  - 列出目录内容
  - 创建/删除文件/目录

- [F2.3] 环境管理
  - 设置环境变量
  - 设置工作目录
  - 设置进程超时

**验收标准**：
- 进程启动延迟 < 50ms
- 输出流延迟 < 100ms
- 文件传输速度 > 50MB/s

---

#### F3: E2B SDK 兼容层

**用户故事**：
> 作为现有 E2B 用户，我希望只修改 API endpoint 就能无缝迁移到 gVisor E2B。

**功能点**：
- [F3.1] TypeScript SDK
  - 与 E2B SDK 相同的类名、方法名、参数
  - `Sandbox.create()`, `sandbox.betaPause()`, `sandbox.kill()` 等

- [F3.2] Python SDK
  - 与 E2B Python SDK API 一致

- [F3.3] REST API
  - 完全符合 E2B OpenAPI 规范
  - 相同的请求/响应格式
  - 相同的错误码

**验收标准**：
- 通过 E2B SDK 测试套件（修改 endpoint 后）
- 100% API 覆盖率

## 10. 技术选型依据

### 10.1 gVisor vs Firecracker

| 维度 | gVisor | Firecracker |
|------|--------|-------------|
| **隔离层次** | 系统调用 | 虚拟机 |
| **启动速度** | 50-100ms | 150-300ms |
| **内存开销** | 10-20MB | 50-100MB |
| **部署要求** | Docker 即可 | 需要 KVM |
| **K8s 集成** | 原生支持 | 需要定制 |

**选择 gVisor 的理由**：
- 更低的运营成本（资源和部署）
- 更好的容器生态兼容性
- 足够的安全隔离（系统调用级别）

### 10.2 FastAPI vs Go HTTP

**选择 FastAPI (Python)** 作为控制平面：
- 快速开发和迭代
- 丰富的生态系统（ORM、异步库）
- OpenAPI 自动生成
- 适合复杂业务逻辑

**选择 Go** 作为 envd 数据平面：
- 高性能（低延迟 RPC）
- 低内存占用（每个沙盒内运行）
- 成熟的 gRPC/Connect 库
- 易于静态编译和分发

### 10.3 PostgreSQL vs MySQL

**选择 PostgreSQL**：
- 更强的 JSON 支持（metadata 存储）
- 更好的并发性能（MVCC）
- 更丰富的数据类型（枚举、数组）
- 更活跃的开源社区
